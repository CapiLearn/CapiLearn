# Retrospective

## What we planned at the Hackathon

At the Hackathon, our plan was to build a small but useful AI learning assistant. The core scope was a RAG chat experience grounded in course material, with guardrails to keep the assistant focused on learning.

We discussed several supporting ideas early, including conversation search, student activity tracking, and retrieved-context visibility. The instructor dashboard was part of the required product scope, but it depended on more careful schema design and analytics work than the RAG and guardrail pieces. The MVP had to support both the student chat experience and the instructor-facing workflow, even though the dashboard details were less concrete at the start.

## What changed and why

The biggest change was that the project became more platform-like than we expected. To make the assistant usable across student, instructor, and admin flows, we had to spend significant time on authentication and role aware routing. Those pieces were part of making the required product work across roles and to be deployment ready.

Our RAG implementation also changed. We started with Chroma because it was quick to prototype, local, and cheap to run while the deployment requirements were still being finalized. Later, we moved retrieval data into PostgreSQL with pgvector. That was a better fit for the rest of the app because conversations, users, activity, and retrieval data could all live behind the same database and migration workflow.

The instructor dashboard became a required feature with room for growth and expansion. We added protected instructor dashboard APIs and a frontend dashboard view, but the frontend still uses demo data in places instead of being fully wired to live analytics.

We also made retrieved course context visible so that users could trust the answer generated by the LLM. The UI citation feature helped connect each answer back to the course material that supported it.

## What we learned

We learned that the data model needed to be defined earlier and treated as part of the MVP. That would have made it easier to create new features on top of the schemas.

We also learned to choose core infrastructure based on clearer project goals. Chroma helped us move quickly at first, and the later move to PostgreSQL and pgvector gave us better deployment consistency and long-term maintainability.

Our estimates were too optimistic. Integrating the various services and getting things deployment ready took longer than expected. We learned to plan more buffer for integration work.

Finally, we learned that access to actual users should shape how feedback is planned. Since we did not have meaningful access to real users during the project, clearer reviewer-facing success criteria would have helped guide product decisions earlier.
